ТЕХНИЧЕСКОЕ ЗАДАНИЕ (ТЗ)Разработка веб-сайта для кафе бурятской кухни «Байкал Буузы» (Версия 3.0, Финальная)1. Общие сведенияНастоящее техническое задание определяет требования к разработке коммерческого веб-сайта для кафе «Байкал Буузы». Продукт представляет собой полноценную экосистему для клиентов заведения, обеспечивающую ознакомление с актуальным меню бурятской кухни, оформление онлайн-заказов на доставку, авторизацию, интеграцию с платежным шлюзом и бронирование посадочных мест в зале.2. Стек технологийВыбор стека обусловлен высокими требованиями к производительности бэкенда, отказоустойчивости и необходимости обеспечения бесшовного пользовательского опыта (UX) на стороне клиента.Бэкенд: Go (Golang). Используется пакет crypto/bcrypt для хэширования паролей, сессии реализуются на базе безопасных HTTP-only cookies (или защищенных JWT).Фронтенд: SvelteKit (Svelte) + TailwindCSS для быстрой, современной и адаптивной верстки.База данных: PostgreSQL — реляционная СУБД для надежного хранения данных.Платежный шлюз: API ЮKassa для обработки онлайн-платежей и приема вебхуков.3. Требования к кодовой базе и архитектуре3.1. Архитектурные паттерны и чистый кодРазделение на слои (Clean / Layered Architecture): Код бэкенда строго структурируется на независимые слои:Handlers (Слой контроллеров/транспорта) — осуществляет прием входящих HTTP-запросов, парсинг JSON/форм и первичную валидацию.Services (Слой бизнес-логики) — инкапсулирует ключевые правила системы (расчет стоимости, алгоритмы повтора заказов, валидация броней).Repositories (Слой доступа к данным) — изолирует прямые SQL-запросы к PostgreSQL.Соблюдение принципов DRY и SOLID: Исключается дублирование кода. Каждый модуль выполняет единственную зону ответственности. Компоненты фронтенда декомпозируются на атомарные UI-элементы.DTO и Маппинг: Для передачи данных между слоями и изоляции внутренних структур БД от внешнего API в обязательном порядке применяются структуры DTO (Data Transfer Objects). Конвертация сущностей реализуется явными методами-мапперами.Запрет на комментарии: В соответствии со стандартами самодокументируемого кода, в кодовой базе полностью отсутствуют комментарии. Логика программы должна быть очевидной благодаря понятным, лаконичным и точным именам переменных, функций, структур и методов.3.2. БезопасностьЗащита от SQL-инъекций: Строгий запрет на конкатенацию строк при формировании SQL-запросов. Все операции с базой данных используют исключительно параметризованные запросы с плейсхолдерами (например, $1, $2 в PostgreSQL).Защита от XSS и CSRF: Данные авторизации (токены сессий) передаются и хранятся на клиенте исключительно в HTTP-only и Secure файлах cookie. Вся пользовательская информация, выводимая на фронтенде, экранируется средствами SvelteKit.Хэширование данных: Хранение паролей в открытом виде запрещено. Используется криптостойкий алгоритм адаптивного хэширования bcrypt.3.3. Требование к логированиюВнутренний системный текстовый вывод программного обеспечения (системные сообщения, логи работы, ошибки времени выполнения и отладочная информация бэкенда на Go) должен осуществляться исключительно строчными (маленькими) русскими буквами.Примеры: "получен запрос на регистрацию пользователя", "ошибка: такой номер телефона уже существует".4. Функциональные требования4.1. Аутентификация и авторизацияРегистрация: Упрощенная форма ввода данных без SMS-верификации. Обязательные поля: Имя, Телефон (выступает уникальным логином в системе), Пароль. При регистрации бэкенд проверяет уникальность телефона в БД.Авторизация: Вход в систему по номеру телефона и паролю. При успешном совпадении хэшей сервер выдает защищенную сессию.4.2. Каталог продукции и фильтрацияОтображение блюд в виде визуальной сетки карточек. Каждая карточка содержит: качественное изображение, наименование, краткое описание состава, вес/объем, стоимость и эргономичную кнопку быстрого добавления в корзину.Интерактивный фильтр по категориям меню (Буузы, Супы, Салаты, Десерты, Напитки). Переключение категорий должно происходить мгновенно без полной перезагрузки страницы.4.3. Детальная страница товараКаждое блюдо имеет уникальный статический URL (роутинг SvelteKit).На странице выводится расширенное описание: детальный состав, аллергены, КБЖУ (калорийность, белки, жиры, углеводы), увеличенная фотография высокого разрешения и селектор изменения количества единиц товара для добавления в корзину.4.4. Корзина (Модальное окно)Глобальный компонент, вызываемый кликом на иконку в шапке сайта на любой странице. Реализуется через реактивные сторы (Svelte stores).Возможность прямого инкремента/декремента количества порций (+/-) и удаления позиций с мгновенным реактивным пересчетом итоговой суммы заказа.4.5. Оформление заказа и доставкаПереход из корзины на выделенный экран оформления заказа.Для авторизованных пользователей: Автоматическое предзаполнение полей «Имя», «Телефон» и «Сохраненный адрес» из личного профиля.Для неавторизованных пользователей: Доступно оформление заказа в режиме «Гость» с обязательным ручным заполнением полей (Имя, Телефон, Адрес).Выбор способа оплаты: «Онлайн-оплата (ЮKassa)» или «При получении наличными/картой».4.6. Личный кабинет пользователя и историяРаздел профиля: Возможность просмотра и изменения персональной информации, включая менеджмент дефолтного адреса доставки.История заказов: Табличный или списочный вывод всех заказов с указанием даты, перечня блюд, статуса оплаты и статуса выполнения (Новый, Оплачен, Готовится, В пути, Доставлен, Отменен).Функция «Повторить заказ»: Кнопка в истории, позволяющая в один клик очистить текущую корзину и наполнить её идентичным составом позиций из выбранного исторического заказа.История бронирований: Отображение текущего статуса зарезервированных столов клиента.4.7. Интеграция с ЮKassaВ случае выбора онлайн-оплаты бэкенд инициирует транзакцию через ЮKassa API и возвращает URL для редиректа на защищенную платежную страницу.Бэкенд предоставляет публичный эндпоинт для приема Webhook-уведомлений от ЮKassa. При получении события об успешном списании средств статус заказа в базе данных автоматически переводится в "оплачен".4.8. Бронирование столикаПубличная интерактивная веб-форма с полями: Имя, Телефон, Дата, Желаемое время, Количество персон, Текстовый комментарий (пожелания).В случае, если форма заполняется авторизованным пользователем, бронь автоматически линкуется с его аккаунтом в базе данных для отображения в ЛК.5. Нефункциональные требованияАдаптивность интерфейса: Интерфейс, созданный с помощью TailwindCSS, должен корректно и без потери читаемости отображаться на мобильных телефонах, планшетах и широкоформатных мониторах (Mobile First подход).Защита роутов (Route Guards): Клиентские маршруты личного кабинета (на стороне SvelteKit) должны проверять наличие авторизации и перенаправлять неавторизованных посетителей на страницу входа.
